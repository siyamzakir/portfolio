Most AI chatbot integrations are duct-tape jobs. You have a CRM, someone adds a third-party chatbot widget, and you spend the next six months syncing data between two systems that were never meant to talk.

At [AppsZone](https://appszonebd.com), I took a different approach: the chatbot is **part of the CRM**, not bolted onto it.

## The Problem with Bolt-On Bots

When you plug a chatbot into an existing system as an integration, you inherit all the friction:

- Data lives in two places
- Leads need manual handoff from bot to agent
- Context is lost between conversations
- You pay two vendors instead of one

The agents using our immigration CRM were spending more time copy-pasting between the chatbot dashboard and the CRM than actually processing cases.

## The Architecture

I built the chatbot as a first-class module inside the Laravel + React CRM, not a standalone service.

```php title="ChatbotController.php"
class ChatbotController extends Controller
{
    public function handle(ChatRequest $request): JsonResponse
    {
        $lead = Lead::firstOrCreate(
            ['session_id' => $request->session_id],
            ['source' => 'chatbot', 'status' => 'new']
        );

        $response = OpenAI::chat()->create([
            'model' => 'gpt-4o',
            'messages' => $this->buildContext($lead, $request->message),
        ]);

        $lead->conversations()->create([
            'message' => $request->message,
            'reply'   => $response->choices[0]->message->content,
        ]);

        if ($this->shouldEscalate($response)) {
            $lead->update(['status' => 'qualified', 'assigned_to' => $this->nextAvailableAgent()]);
            event(new LeadQualified($lead));
        }

        return response()->json(['reply' => $response->choices[0]->message->content]);
    }
}
```

Every conversation is stored directly on the `Lead` model. No sync job. No webhook. No eventual consistency headaches.

## What Changed for Agents

Before: agent opens CRM, opens chatbot dashboard, reads conversation, copy-pastes to CRM notes, assigns lead manually.

After: lead appears in the CRM automatically, conversation is already attached, the bot has already qualified intent and set the status. The agent just acts.

Response time dropped by **80%**. Manual triage work dropped to near zero.

## The OpenAI Prompt Layer

The system prompt is built dynamically from the lead's current CRM state:

```typescript title="buildContext.ts"
function buildContext(lead: Lead, newMessage: string): Message[] {
  return [
    {
      role: "system",
      content: `You are a visa application assistant for an Australian migration agency.
The user is interested in: ${lead.visa_type ?? "not specified"}.
Their current application stage: ${lead.stage}.
Keep answers concise and always ask for one missing document at a time.`,
    },
    ...lead.conversations.map((c) => [
      { role: "user", content: c.message },
      { role: "assistant", content: c.reply },
    ]).flat(),
    { role: "user", content: newMessage },
  ];
}
```

The bot knows exactly where the applicant is in the pipeline and responds accordingly.

## Lessons

- **Integrate at the data layer, not the API layer.** If two features share a database, they're one product.
- **Let the bot write to the CRM directly.** Don't sync — own.
- **Escalation logic is the product.** The AI part is almost trivial once you get the escalation right.
