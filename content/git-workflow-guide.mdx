In 2014 when I started on Upwork, deploying meant zipping files, FTP-ing to shared hosting, and crossing my fingers. Today I run automated pipelines that build, test, and deploy to Docker in under 3 minutes — with a rollback button if anything breaks.

Here's the exact workflow I landed on after iterating through every bad idea in between.

## Branch Strategy (Keep It Simple)

```
main         ← production, always deployable
staging      ← pre-production testing
feature/*    ← one feature per branch
fix/*        ← hotfixes, merged to main + staging
```

Three rules:
1. Nothing merges to `main` without a PR review.
2. All PRs must pass CI before merge.
3. `main` deploys automatically. `staging` deploys automatically. Everything else is manual.

## Commit Message Convention

```bash
feat: add visa application status webhook
fix: resolve lead duplicate on chatbot re-entry
chore: upgrade Laravel to 11.x
refactor: extract email notification into job
docs: add API endpoint docs for /leads
```

This feeds into automatic changelogs and makes `git log --oneline` actually readable.

## The GitHub Actions Pipeline

```yaml title=".github/workflows/ci.yml"
name: CI

on:
  pull_request:
    branches: [main, staging]

jobs:
  test:
    runs-on: ubuntu-latest

    services:
      mysql:
        image: mysql:8.0
        env:
          MYSQL_DATABASE: test_db
          MYSQL_ROOT_PASSWORD: root
        options: --health-cmd="mysqladmin ping" --health-interval=10s

      redis:
        image: redis:alpine

    steps:
      - uses: actions/checkout@v4

      - name: Setup PHP
        uses: shivammathur/setup-php@v2
        with:
          php-version: "8.3"
          extensions: pdo_mysql, redis, bcmath, gd

      - name: Install dependencies
        run: composer install --prefer-dist --no-progress

      - name: Run tests
        env:
          DB_CONNECTION: mysql
          DB_HOST: 127.0.0.1
          DB_DATABASE: test_db
          DB_USERNAME: root
          DB_PASSWORD: root
          REDIS_HOST: 127.0.0.1
        run: php artisan test --parallel
```

## Zero-Downtime Deploy

```yaml title=".github/workflows/deploy.yml"
name: Deploy to Production

on:
  push:
    branches: [main]

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Deploy via SSH
        uses: appleboy/ssh-action@v1
        with:
          host: ${{ secrets.PROD_HOST }}
          username: deploy
          key: ${{ secrets.DEPLOY_KEY }}
          script: |
            cd /var/www/appszonebd
            git pull origin main
            docker compose build app
            docker compose up -d --no-deps --scale app=2 app
            sleep 10
            docker compose up -d --no-deps --scale app=1 app
            docker compose exec -T app php artisan migrate --force
            docker compose exec -T app php artisan queue:restart
            docker compose exec -T app php artisan config:cache
```

The scale-to-2 then scale-to-1 trick gives a brief overlap during container restart — poor man's blue-green deployment without a load balancer.

## Rollback in 30 Seconds

```bash
# On the server
git log --oneline -5          # find last good commit
git checkout <commit-hash>
docker compose up -d --build
```

The database migration issue is the only real risk. That's why migrations must be backward-compatible. Never drop a column in the same deploy that removes it from the app.
